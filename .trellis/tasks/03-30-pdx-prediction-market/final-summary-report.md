# PDX 项目最终收敛报告

> 基于 7 轮讨论 + 外部调研的收敛结论

---

## 一、项目演进路径

```
讨论1: 完整预测市场平台（4层架构，智能合约+AI仿真+算力+反内幕）
  ↓ 发现问题：MiroFish 质量未验证，零和游戏无法兑现"AI赚钱"
讨论2: 诚实分析 → 鲸鱼操纵只能缓解、内幕交易无法解决、不可能三角
  ↓ 发现问题：核心假设不成立
讨论3: 仿真业务需求 → 保险/金融/咨询 → 但质量未验证前无法推进
  ↓ 发现问题：先花$500验证质量
讨论4: MiroFish 质量验证方案设计 → Brier Score 对比实验
  ↓ 用户反馈：不做预测，做证伪
讨论5: 换起点 → 从"闲置算力变现"出发 → 信号市场（卖事实不卖预测）
  ↓ 用户确认：回归预测市场，但用 topic+证伪 模式
讨论6: 证伪市场设计 → 命题生命周期 + 多智能体证伪流水线 + 评级系统
  ↓ 工程问题：重复数据过滤
讨论7: 去重设计 → 4层去重 + 3阶段搜索 + 领域bot路由
  ↓ 收敛
最终: ClawHub pdx-verify 技能 → 最低成本验证 → 渐进扩展
```

### 确认可行的

```
1. AI 做信息搜索和验证 → LLM+RAG 的强项，已验证能力
2. 领域 bot 做专业化搜索 → OpenClaw 已有基础
3. 证伪比预测更可衡量质量 → 证据链可验证
4. 闲置算力跑搜索任务 → CPU 即可，不需要 GPU
5. ClawHub 技能作为分发渠道 → 340K 用户即刻可触达
6. 技能开发成本极低 → SKILL.md + 脚本目录，无需后端
```

### 确认不可行的

```
1. "AI自动赚钱"承诺 → 零和游戏 + 不可能三角
2. MiroFish 大规模仿真预测 → 质量未验证，可能是"贵的噪音"
3. 消除内幕交易 → 知道真实结果的人永远有优势
4. 1000个AI智能体 = 更好的预测 → 无证据支持
5. 从零搭建完整平台 → 成本高、周期长、获客难
```

## 三、收敛方案：ClawHub pdx-verify 技能

### 核心洞察

```
旧思路：从零搭建 → Web2 平台 → 获客 → 验证 → 可能上链
  问题：需要用户、需要基础设施、需要营销、周期长

收敛方案：做一个 ClawHub 技能 → 340K 用户池即刻可用 → 验证 → 扩展
  本质：把证伪引擎做成 OpenClaw 生态的一个"插件"

类比：
  旧思路 = 自己开一家餐厅
  新思路 = 在美团上开一个外卖店
```

### 验证后的技能架构

```
~/.openclaw/skills/pdx-verify/
├── SKILL.md              # 核心：证伪引擎指令
│   ├── frontmatter:
│   │   name: pdx_verify
│   │   description: 对任何命题进行结构化证伪，输出证伪报告和评级
│   │   user-invocable: true
│   │   metadata.openclaw.requires.bins: []   # 无外部依赖
│   │   metadata.openclaw.requires.env: []    # 无API key需求
│   └── body:
│       ├── 命题解析指令（提取可验证声明）
│       ├── 多角度搜索策略（英文/中文/行业源）
│       ├── 证据评估框架（来源可信度×相关度×时效性）
│       ├── 去重逻辑（URL去重 + 语义去重）
│       └── 报告生成模板（A/B/C/D/F 评级）
├── scripts/
│   └── parse_proposition.py  # 命题结构化提取辅助脚本
└── references/
    └── credibility_sources.md  # 来源可信度参考表
```

### 用户体验

```
用户安装：clawhub install pdx-verify （一键）

使用方式：
  用户：/pdx-verify "台积电3nm良率已达80%"

AI智能体自动执行：
  1. 解析命题 → 提取可验证声明
  2. 多角度搜索反证（英文、中文、行业源）
  3. 评估每条反证的可信度和力度
  4. 输出结构化证伪报告

输出示例：
  ┌─────────────────────────────────────────────┐
  │ PDX 证伪报告：台积电3nm良率已达80%           │
  │                                             │
  │ 评级：C（中等置信度 — 发现有力反证）          │
  │                                             │
  │ 反证 1（强）：                               │
  │   DigiTimes 2026/2/15 报道良率约70-75%       │
  │   来源可信度：7/10                           │
  │                                             │
  │ 反证 2（中）：                               │
  │   三星3nm良率60%推算台积电约70-75%            │
  │   间接推理，非直接数据                        │
  │                                             │
  │ 支持证据 1：                                 │
  │   TSMC 2025 Q4 财报提到"良率持续改善"         │
  │   但未给具体数字                              │
  │                                             │
  │ 结论：该命题可能不准确，最可靠数据指向70-75%   │
  └─────────────────────────────────────────────┘
```

### 发布流程（已验证）

```
1. 创建 SKILL.md 及目录结构
2. 本地测试：openclaw agent --message "/pdx-verify '测试命题'"
3. 安装 ClawHub CLI：npm install -g clawhub
4. 登录：clawhub login （需 GitHub 账号，≥1周）
5. 发布：clawhub publish --name pdx-verify --version 0.1.0
6. 340K 用户可通过 ClawHub 发现并安装
```

## 五、竞品分析

### 为什么"结构化证伪"是空白赛道

```
现有工具                     做什么          不做什么
─────────────────────────────────────────────────────
Perplexity / ChatGPT       回答问题        不做对抗性证伪
Snopes / PolitiFact         人工 fact-check  不可编程，慢
Google Fact Check API       索引已有 check   不生成新 check
ClaimBuster                 检测可查声明     不执行验证
Logically                   检测虚假信息     企业级，非开发者
UMA Oracle                  代币投票验证     不用 AI 证伪
Supra AI Oracle             多AI委员会       专注链上数据验证
Chainlink                   价格数据         不处理主观事件

PDX pdx-verify             结构化证伪        ← 空白赛道
  - 主动搜索反证（多角度）
  - 证据评分和分级
  - 对抗性"红队"搜索
  - 可编程（技能格式）
  - 去中心化（跑在每个用户的OpenClaw上）
```

### 关键差异点

```
1. "回答" vs "证伪"
   Perplexity："台积电3nm良率是多少？" → 综合搜索结果给答案
   pdx-verify："台积电3nm良率已达80%" → 主动寻找反证攻击这个命题

2. "人工" vs "自动化"
   Snopes：记者花几天写一篇 fact-check
   pdx-verify：AI 几分钟内从多角度搜索+分析+出报告

3. "中心化" vs "去中心化"
   Logically：一家公司的服务器
   pdx-verify：340K 用户的 OpenClaw 实例 = 天然分布式

4. "被动" vs "主动"
   Google Fact Check API：等人类 fact-checker 发布后索引
   pdx-verify：用户提命题，AI 立即主动搜证伪
```

## 六、可行性评估

### 五维度评分

```
1. 技术可行性：⭐⭐⭐⭐⭐
   - SKILL.md 格式已验证，有明确规范
   - 不需要后端/数据库/基础设施
   - OpenClaw 内置搜索能力，只需编排
   - 一个开发者 2 周可完成 MVP

2. 市场可行性：⭐⭐⭐⭐
   - 340K OpenClaw 用户 = 现成技术用户群
   - "这条信息可靠吗？" = 普遍需求
   - AI 验证市场 $2.1B（2026）
   - UMA/Polymarket 在做类似方向 = 需求被验证
   - 全球无现有"结构化证伪"产品 = 空白赛道

3. 经济可行性：⭐⭐⭐⭐⭐
   - Step 1 总成本 $200 → 几乎零风险
   - 如果失败 → 损失 2-4 周 + $200
   - 每一步都有明确 go/no-go 标准

4. 竞争可行性：⭐⭐⭐⭐⭐
   - 全球无同类产品
   - Perplexity/ChatGPT 做回答不做证伪
   - Snopes 做人工不做自动化
   - ClawHub 无证伪类技能

5. 生态利用度：⭐⭐⭐⭐⭐
   - 完美利用闲置算力（跑在用户设备上）
   - 完美利用 ClawHub 分发（不需自建获客）
   - 天然去中心化（每个 OpenClaw 实例独立运行）
```

### 风险及缓解

```
风险 1：ClawHub 审核不通过
  缓解：先在 GitHub 发布，用户手动安装
  概率：低

风险 2：用户不安装/不使用
  缓解：做几个高质量证伪案例作为 demo
  止损：$200 + 2周

风险 3：证伪质量不够好
  缓解：先限制命题类型（科技/商业，不做政治/主观）
  迭代：根据反馈持续优化 SKILL.md 指令

风险 4：被大厂抄
  缓解：先发优势 + 众包证伪数据的网络效应
  即使被抄，用户贡献的证伪数据不容易复制
```
